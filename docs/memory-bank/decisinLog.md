# Decisin Log

## DL-001: Keep the site on Bedrock
- **Status**: accepted
- **Context**: the repository already uses Bedrock layout and configuration.
- **Decision**: keep Bedrock as the platform model for the site.
- **Why**: it matches the existing repository structure and deployment assumptions.
- **Consequences**: documentation and tooling assume `web/app` is the content root.

## DL-002: Keep the theme frontend on classic PHP templates
- **Status**: accepted
- **Context**: the shipped theme uses `theme.json` and custom blocks, but page rendering is driven by PHP templates.
- **Decision**: treat the theme as a classic-template frontend with Gutenberg-era extensions.
- **Why**: this matches the current implementation and keeps rendering predictable.
- **Consequences**: frontend work starts from PHP templates first, not block templates.

## DL-003: Keep design tokens centralized in `src/tokens/design-tokens.json`
- **Status**: accepted
- **Context**: theme token outputs are generated by `scripts/sync-tokens.mjs`.
- **Decision**: use `design-tokens.json` as the canonical source for theme tokens.
- **Why**: it prevents drift between `theme.json`, runtime CSS variables, and Tailwind config.
- **Consequences**: generated token outputs should not be edited manually.

## DL-004: Register custom blocks from `build/`
- **Status**: accepted
- **Context**: `inc/blocks.php` registers blocks from built metadata directories.
- **Decision**: keep `src/blocks/*` as source only and `build/blocks/*` as runtime registration targets.
- **Why**: built metadata and asset paths need to match compiled outputs.
- **Consequences**: new blocks require a successful build before they are usable.

## DL-005: Treat the theme as the main custom artifact
- **Status**: accepted
- **Context**: most custom user-facing behavior is contained in `web/app/themes/proger-blog`.
- **Decision**: document and evolve the theme as a standalone artifact within the repository.
- **Why**: it creates a clear extension model for future blocks and UI work.
- **Consequences**: theme documentation should stay deeper than general project documentation.
